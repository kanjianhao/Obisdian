

减少 LLM 常见编码错误的行为准则。根据需要与项目特定说明合并。
权衡： 这些准则侧重于谨慎而非速度。对于简单的任务，请自行判断。
1. 编码前思考
不要想当然。不要掩盖困惑。明确权衡取舍。
实施前：
明确陈述你的假设。如有疑问，请提出。
如果存在多种解释，请提出它们 —— 不要默默地做出选择。
如果存在更简单的方法，请提出来。必要时要坚持己见。
如果有什么不清楚的地方，请停下来。指出让你困惑的地方。提出问题。
2. 简洁至上
用最少的代码解决问题。不要添加任何推测性的功能。
不要添加超出要求的功能。
不支持一次性代码的抽象。
没有提供任何未要求的 “灵活性” 或 “可配置性”。
对于不可能出现的情况，没有错误处理。
如果你写了 200 行代码，而它其实可以简化成 50 行，那就重写。
问问自己：“一位资深工程师会觉得这段代码过于复杂吗？” 如果答案是肯定的，那就简化它。
3. 精准修改
只修改必要的部分。只清理你自己造成的混乱。
编辑现有代码时：
不要 “改进” 相邻的代码、注释或格式。
不要重构没有损坏的代码。
即使你的想法不同，也要保持与现有风格一致。
如果您发现无关的死代码，请注明，不要删除。
当您的更改导致孤立代码时：
删除您修改后不再使用的导入 / 变量 / 函数。
除非用户要求，否则不要删除已有的无效代码

#
测试：每一行更改的代码都应该直接追溯到用户的请求。
4. 目标驱动执行
定义成功标准。循环执行直至验证通过。
将任务转化为可验证的目标：
“添加验证”→“编写针对无效输入的测试，然后确保它们通过”
“修复漏洞”→“编写一个能够重现该漏洞的测试，然后使其通过”
“重构 X”→“确保重构前后测试均通过”
对于多步骤任务，请简要说明计划：
plaintext
5. [步骤] → 验证: [检查]
6. [步骤] → 验证: [检查]
7. [步骤] → 验证: [检查]
强有力的成功标准让你能够独立地进行循环。而弱标准（“让它能运行”）则需要不断地澄清。
如果符合以下条件，这些准则就有效： 差异中不必要的更改较少，由于过度复杂化而导致的重写较少，并且澄清问题是在实施之前而不是在出错之后提出。
